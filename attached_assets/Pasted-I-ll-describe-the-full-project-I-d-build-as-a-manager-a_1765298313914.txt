## 1. Product Overview

- **Name**: Udaan Document Management System (Enterprise Edition)  
- **Goal**: Central system for uploading, organizing, searching, securing, and auditing documents for an organization (or multiple organizations/tenants).  
- **Target**: Internal company users (admin, managers, staff) managing large volumes of documents with strict access control and auditability.

---

## 2. Tech Stack (No Docker Locally)

- **Monorepo root**: `udaan-dms/`
- **Backend**:
  - Node.js (>=18), TypeScript
  - Framework: Express or Fastify (Express is fine)
  - Database: PostgreSQL
  - ORM: Prisma or TypeORM (I’d choose Prisma)
  - Auth: JWT + refresh tokens
- **Frontend**:
  - React + TypeScript
  - Vite for bundling
  - UI: TailwindCSS + Headless UI or similar
  - React Router for routing
- **Local Dev**:
  - No Docker. Use:
    - `npm run dev:backend` (starts API on e.g. `http://localhost:4000`)
    - `npm run dev:frontend` (starts UI on e.g. `http://localhost:5173`)
  - PostgreSQL installed locally (or remote dev DB), configurable via `.env`.

## 4. Core Features / Modules

- **Organizations & Tenancy**
  - Multiple organizations (tenants).
  - Users belong to one organization.
- **Users & Roles**
  - Roles: `SUPER_ADMIN`, `ORG_ADMIN`, `MANAGER`, `STAFF`, `VIEWER`.
  - Role-based access control for documents, folders, and actions.
- **Documents**
  - Upload (single, bulk).
  - Metadata: title, description, tags, type, createdBy, organizationId, folderId, custom fields.
  - Versioning (optional for Phase 2).
  - Soft delete / archive.
- **Folders**
  - Hierarchical tree (folder with `parentFolderId`).
- **Tags & Metadata**
  - System tags + custom tags.
  - Custom metadata fields configured per organization (Phase 2+).
- **Search**
  - Filter by title, tags, uploader, date range, folder, organization.
  - Simple text search on title/description (no heavy full-text in Phase 1).
- **Workflows (Phase 2)**
  - Review/approval states (`DRAFT`, `UNDER_REVIEW`, `APPROVED`, `REJECTED`).
- **Audit Logging**
  - Actions tracked: login, upload, download, update metadata, delete, restore, permission changes.
- **Security**
  - JWT auth, refresh tokens.
  - Password hashing (bcrypt).
  - Per-tenant isolation via `organizationId` in all queries.
  - Role-based middleware on routes.

## 6. Backend API Design (High Level)

- **Auth**
  - `POST /api/auth/register` (for initial super admin seed, then org admin only)
  - `POST /api/auth/login`
  - `POST /api/auth/refresh`
  - `POST /api/auth/logout`
- **Organizations**
  - `GET /api/organizations` (super admin)
  - `POST /api/organizations`
  - `GET /api/organizations/:id`
- **Users**
  - `GET /api/users` (within org)
  - `POST /api/users`
  - `PATCH /api/users/:id`
  - `DELETE /api/users/:id` (soft)
- **Folders**
  - `GET /api/folders/tree`
  - `POST /api/folders`
  - `PATCH /api/folders/:id`
  - `DELETE /api/folders/:id`
- **Documents**
  - `GET /api/documents` (with query filters)
  - `GET /api/documents/:id`
  - `POST /api/documents` (multipart upload)
  - `PATCH /api/documents/:id`
  - `DELETE /api/documents/:id` (soft delete)
  - `POST /api/documents/:id/restore`
  - `GET /api/documents/:id/download`
- **Search**
  - `GET /api/search/documents` (q, tags, dateFrom, dateTo, folderId, etc.)
- **Audit**
  - `GET /api/audit/logs` (filter by user, action, date range)

All routes protected by JWT auth except login/register; use org + role checks in middleware.
